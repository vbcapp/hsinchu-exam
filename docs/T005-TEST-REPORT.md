## 📋 功能摘要

實作 `getHourlyEfficiency(userId)` API，分析用戶在不同時段（0-23 時）的答題效率。

### 回傳資料格式

```javascript
{
  success: true,
  data: {
    hourlyStats: [
      { hour: 7, correct: 15, total: 18, accuracy: 83 },
      { hour: 8, correct: 20, total: 22, accuracy: 91 },
      // ... 每小時統計
    ],
    bestHour: 8,               // 最佳時段
    worstHour: 23,             // 最差時段
    bestAccuracy: 91,          // 最高正確率
    worstAccuracy: 45,         // 最低正確率
    recommendation: '建議在早上 8-9 點答題，該時段正確率達 91%，表現最佳'
  }
}
```

---

## 🎯 實作細節

### 1. 資料來源
- **資料表：** `answer_records`
- **查詢欄位：** `is_correct`, `created_at`
- **時區處理：** 使用 JavaScript `Date.getHours()` 自動處理本地時區

### 2. 核心邏輯
1. 查詢用戶所有答題記錄
2. 從 `created_at` 提取小時（0-23）
3. 按小時分組統計正確率
4. 找出正確率最高/最低時段（至少 5 題以上）
5. 生成個性化建議文字

### 3. 建議生成規則
- **最佳時段：** 正確率比平均高 10% 以上 → 建議在該時段答題
- **最差時段：** 正確率比平均低 15% 以上 → 建議避免該時段
- **數據不足：** 某時段少於 5 題 → 不納入最佳/最差計算
- **新用戶：** 無答題記錄 → 回傳友善提示訊息

## 📊 效能考量

### 查詢效能
- **查詢複雜度：** O(n)，其中 n 是答題記錄總數
- **預期回應時間：** < 500ms（假設答題記錄 < 10,000 筆）
- **資料庫負載：** 單次 SELECT 查詢，無 JOIN

### 優化建議（未來）

1. 新增 `answer_records` 索引：`(user_id, created_at)`

## 🐛 已知限制

1. **時區問題：** 目前使用用戶本地時區，不同時區用戶無法比較
   - **解決方案（未來）：** 統一使用 UTC+8（台灣時區）

2. **數據稀少時的誤判：** 某時段只有 1-2 題可能產生極端正確率
   - **解決方案：** 已實作「至少 5 題」門檻，數據不足不納入最佳/最差計算

3. **無法區分工作日/週末：** 週一早上可能表現差，週日早上可能表現好
   - **解決方案（未來）：** 新增 `getDayOfWeekEfficiency(userId)` API
